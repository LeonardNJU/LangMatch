# 文言文作为 LLM 自然语言压缩提示的开题报告草案

## 1. 题目

文言文能否作为大型语言模型中的自然语言压缩提示：面向 token 效率与任务成功率的系统研究

英文题目：Can Classical Chinese Serve as a Natural Prompt Compression Language for LLMs? A Systematic Study of Token Efficiency and Task Success

## 2. 研究背景与问题提出

大型语言模型的推理成本与上下文长度直接相关。输入越长，通常意味着更高的 token 成本、更高的时延，以及更强的上下文占用压力。现有 prompt compression 研究已经证明：在尽量保持性能的前提下压缩输入，是提升 LLM 使用效率的一条重要路径。

另一条相关文献线指出，不同自然语言在 tokenizer 下并不公平。相同语义在不同语言中会被切分为不同数量的 token，从而产生实际成本差异。与此同时，跨语言 prompting 工作也显示，提示语言不仅影响 token 数，还会影响模型的行为、推理路径和任务表现。

文言文天然具备高度凝练、信息密度高、语法省略强的特点，因此在直觉上可能是一种适合 LLM 的“自然语言压缩提示”。但这个直觉并不能直接推出两个结论：

1. 文言文在 tokenizer 视角下是否真的更省 token。
2. 即便更省 token，是否还能保持任务成功率与指令遵循能力。

因此，本文研究的问题是：在任务语义不变的前提下，仅将 system prompt 或 instruction prompt 从现代汉语改写为语义等价的文言文，是否能够稳定降低 token 消耗，并在性能上保持可接受的水平。

## 3. 研究问题

### RQ1

在语义等价条件下，文言文 system prompt 是否能稳定减少输入 token 数。

### RQ2

文言文提示带来的 token 节省，是否会伴随任务成功率、指令遵循率或准确率下降。

### RQ3

观察到的差异主要来自 tokenizer 切分，还是来自语言风格对模型行为的诱导。

### RQ4

这一现象是否能够跨模型族复现，尤其是开源模型与闭源 API 模型之间是否一致。

## 4. 核心假设

### H1

在中文提示语境下，文言文版本的 system prompt 在多数任务上会比现代汉语版本消耗更少输入 token。

### H2

文言文不会在所有任务上都保持性能。在高约束、强格式控制、复杂指令遵循任务中，文言文可能因省略和歧义造成成功率下降。

### H3

与简单删词的现代汉语压缩相比，文言文可能是一种保留更多语义结构的自然语言压缩方式，因此在同等长度下可能具有更好的质量成本权衡。

## 5. 相关工作

### 5.1 古汉语与文言文 LLM

古汉语已经形成独立的 LLM 研究子方向。C3Bench、TongGu、Fuxi、WenyanGPT 等工作说明：古汉语不是边缘语域，而是可以被单独建模、评测和优化的对象。但这些工作主要关注理解、生成和 benchmark 构建，而非 token 效率或 prompt compression。

### 5.2 跨语言 prompting

跨语言 prompting 研究表明，提示语言、推理语言和任务语言都会影响模型表现。提示语言不是表层改写，而是会改变模型的行为边界、推理路径甚至格式服从性。这为本文提供了直接动机：文言文若作为提示语言发挥作用，可能不只是“更短”，也可能改变模型的输出风格与控制性。

### 5.3 Token tax 与 tokenizer 公平性

多语言 tokenization 存在系统性差异，导致相同信息在不同语言上具有不同成本。相关工作说明：语言本身就是一个影响成本的变量。本文据此将问题从“文言文是否简练”转化为“文言文是否在 tokenizer 加模型耦合下形成更高的信息密度”。

### 5.4 Prompt compression

LLMLingua、Selective Context 及后续综述已证明 prompt compression 的价值。但这些方法主要是算法式压缩，而不是利用自然语言本身的表达密度做压缩。本文的切入点是将文言文视为一种 language-as-compression 的自然语言方案。

## 6. 研究空白与本文定位

现有工作分别证明了四件事：

1. 古汉语可以单独建模与评测。
2. 提示语言会影响模型行为。
3. 不同语言存在 token 成本差异。
4. 压缩 prompt 具有明确价值。

但公开文献尚未把四条线合并成一个可检验问题：文言文能否作为 LLM 中的自然语言压缩提示。本文的定位不是训练新的文言文模型，而是在严格控制变量下评估文言文作为 system prompt 语言的效率与代价。

## 7. 方法设计

### 7.1 基本控制思路

保持任务内容不变，仅改变外层 system prompt 的语言形式。主对照条件如下：

1. Base：英文基础助手提示。
2. ZH-Compact：现代汉语压缩版提示。
3. WY：文言文压缩版提示。

建议使用如下三组 system prompt：

#### Base

You are a helpful assistant. Follow the user's instructions carefully. Except for code and direct quotations, keep explanations and narration concise. Avoid colloquial filler. Eliminate redundancy and aim for compact, clear expression.

#### ZH-Compact

你是一个有帮助的助手。请严格遵循用户要求。除代码和直接指令外，所有解释与叙述都尽量写得简洁。禁止口语化赘述，删繁就简，要求表达精炼、意思清楚。

#### WY

汝为善应人问之助手，当谨循其命。凡码与引文外，释理叙事宜从简。禁绝白话冗词，务求辞约义明。

### 7.2 Pilot 实验设计

Pilot 目标不是直接得出论文级结论，而是先回答三个问题：

1. 文言文 system prompt 是否减少输入 token。
2. 它是否改变指令遵循行为。
3. 它是否损害一般知识与推理准确率。

第一阶段采用 GPT-4o-compatible API，小规模跑通并观察趋势。当前工程支持两类 pilot：

1. IFEval 与 MMLU-Pro 子集：用于观测指令遵循与客观题推理。
2. MATH-500 文本子集：过滤带 asy 图形的题目，使用中英兼容的题面包装，并只评估最终答案正确率。若研究重点转向“解题能力”而非“服从格式要求”，则优先使用这一设置。

### 7.3 指标

输入侧：

1. Prompt tokens
2. Completion tokens
3. Total tokens
4. 相对 Base 的输入 token 压缩率

任务侧：

1. IFEval strict success rate
2. MMLU-Pro accuracy
3. MATH-500 answer accuracy

综合效率侧：

1. Score per 1k tokens
2. Token-normalized success

### 7.4 统计与分析

Pilot 阶段先报告描述统计与配对差值：

1. 各条件平均 prompt tokens
2. 各条件平均 total tokens
3. 各条件 benchmark 分数
4. 文言文相对现代汉语压缩版的质量成本权衡

在正式实验中可进一步采用配对检验和 mixed-effects regression。

## 8. 风险与威胁有效性

### 8.1 语义不等价

文言文改写可能省掉约束项，导致比较不公平。缓解方式是同源改写、人工回译和约束 checklist 对齐。

### 8.2 Prompt 变短但输出变长

输入 token 降低不代表总成本降低，因此必须同时报告 prompt、completion 和 total tokens。

### 8.3 文言文影响的是风格而非压缩

如果 WY 与 ZH-Compact 差异来自“罕见风格诱导”，而非信息密度，则需要加入现代汉语压缩版对照并分开解释。

### 8.4 模型族特异性

首轮 pilot 仅使用 GPT-4o-compatible API，因此结论只能视为单模型族初探。后续重型开源模型实验应放在 `alex` 集群执行，不在本地笔记本运行。
